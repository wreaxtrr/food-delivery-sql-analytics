# Food Delivery SQL Analytics

Учебный SQL-проект, в котором реализована база данных сервиса доставки еды и набор аналитических запросов для расчёта продуктовых и операционных метрик.

В рамках проекта я самостоятельно спроектировал структуру базы данных, настроил связи между таблицами, сгенерировал тестовые данные и провёл их анализ с помощью PostgreSQL.

## Стек

- PostgreSQL
- SQL
- DBeaver

## Данные

В проекте используется синтетический набор данных, имитирующий работу сервиса доставки еды.

Структура базы данных спроектирована на основе типичных процессов такого сервиса:

- регистрация пользователей;
- просмотр ресторанов и товаров;
- добавление товаров в корзину;
- оформление заказа;
- применение промокода;
- оплата заказа;
- назначение курьера;
- приготовление и доставка;
- отмена заказа.

Внешний датасет не использовался. Тестовые данные сгенерированы средствами PostgreSQL с использованием `generate_series()` и `random()`.

Все пользователи, адреса, заказы, платежи и события являются вымышленными и не содержат реальных персональных данных.

## Структура базы данных

База данных содержит следующие таблицы:

- `users` — пользователи сервиса;
- `restaurants` — рестораны;
- `couriers` — курьеры;
- `categories` — категории товаров;
- `products` — товары и блюда;
- `orders` — заказы;
- `order_items` — позиции заказов;
- `payments` — платежи;
- `promocodes` — промокоды;
- `order_status_history` — история изменения статусов заказов;
- `app_events` — события пользователей в приложении.

Между таблицами настроены первичные и внешние ключи. Также добавлены ограничения `NOT NULL`, `UNIQUE`, `CHECK` и значения по умолчанию.

## Аналитические задачи

В проекте рассчитаны следующие показатели:

1. Количество заказов и покупателей по месяцам.
2. GMV и средний чек.
3. Изменение GMV относительно предыдущего месяца.
4. Топ-3 ресторана по выручке в каждом месяце.
5. Доля отменённых заказов по ресторанам.
6. Среднее время доставки.
7. Эффективность курьеров.
8. Сравнение заказов с промокодом и без него.
9. Доля пользователей с повторными заказами.
10. Когортный retention пользователей.
11. RFM-сегментация покупателей.
12. Иерархия категорий товаров.

## Представления

В проекте созданы два представления.

### `v_order_summary`

Содержит подробную информацию о каждом заказе:

- пользователь;
- ресторан;
- курьер;
- промокод;
- количество товаров;
- стоимость товаров;
- размер скидки;
- итоговая сумма;
- способ и статус оплаты;
- время доставки.

### `v_daily_metrics`

Содержит агрегированные метрики по дням:

- количество заказов;
- количество уникальных пользователей;
- GMV;
- средний чек;
- среднее время доставки.

## Запуск проекта

Для запуска проекта необходимо установить PostgreSQL и программу для работы с базой данных, например DBeaver.

SQL-скрипты необходимо выполнить в следующем порядке:

```text
01_create_schema.sql
02_insert_data.sql
03_create_views.sql
05_indexes.sql
```

Файл `01_create_schema.sql` создаёт схему и таблицы.

Файл `02_insert_data.sql` очищает таблицы и генерирует новый набор тестовых данных.

Файл `03_create_views.sql` создаёт представления для дальнейшего анализа.

Файл `05_indexes.sql` добавляет индексы для основных полей.

Запросы из файла `04_analytical_queries.sql` рекомендуется запускать отдельно, так как каждый запрос решает самостоятельную аналитическую задачу.

## Основные результаты анализа

### Влияние промокодов

| Группа | Все заказы | Доставленные заказы | Средний чек доставленного заказа | Выручка |
|---|---:|---:|---:|---:|
| С промокодом | 1 078 | 986 | 4 577,16 | 4 513 084,37 |
| Без промокода | 1 922 | 1 778 | 4 934,32 | 8 773 212,73 |

В рамках сгенерированного набора данных средний чек доставленного заказа с промокодом оказался на 357,16, или примерно на 7,24%, ниже, чем без промокода.

Доля отмен составила около 8,53% для заказов с промокодом и 7,49% для заказов без промокода. Разница составила примерно 1,04 процентного пункта.

Более высокая общая выручка группы без промокода связана в том числе с большим количеством заказов в этой группе, поэтому сравнивать только итоговую сумму выручки недостаточно.

### Продуктовая воронка

| Этап | Количество событий | Конверсия с предыдущего этапа |
|---|---:|---:|
| Открытие приложения | 7 000 | — |
| Просмотр ресторана | 6 011 | 85,87% |
| Просмотр товара | 5 232 | 87,04% |
| Добавление в корзину | 4 382 | 83,75% |
| Начало оформления | 3 714 | 84,76% |
| Создание заказа | 3 000 | 80,78% |

Общая конверсия из открытия приложения в создание заказа составила 42,86%.

Наибольшая относительная потеря пользователей происходит между началом оформления и созданием заказа: до заказа доходит 80,78% пользователей этого этапа. Это означает потерю 19,22% пользователей на последнем шаге воронки.

Следующим потенциальным направлением анализа может быть изучение причин отказа при оформлении заказа: стоимость доставки, способ оплаты, ошибки интерфейса или изменение итоговой стоимости.
